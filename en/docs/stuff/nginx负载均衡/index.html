nginx负载均衡

前言#

nginx的负载均衡，有一次被问到了，以前只听说过，没接触过，这两天学了docker相当于多了好多台server，简单搜索了一下部署了一下负载均衡。

负载均衡的策略#

负载均衡有很多策略，默认的轮询策略，还有其他如随机，分配权重，ip哈希等等。

  • round robin 轮询
  • random 随机
  • weight 权重
  • fair 按响应时长
  • url_hash url的哈希值
  • ip_hash ip的hash值
  • least_hash 最少连接数

常用策略及配置#

第一步，先开几个nginx服务。总共用了三台机器，自己的本机做总机来转发，腾讯云一台机器：开两个docker部署nginx，阿里云一台机器，开一个docker部署nginx。

代码类似下面

# -d表示后台运行# --name为给这个容器的命名# -p 表示端口映射docker run -d --name nginx01 -p 9090:80 nginx

后面就是在主机的nginx的配置文件添加内容。

1.轮询#

upstream nginxs {    server 1.2.3.1:8080;  server 1.2.3.2:8080;    server 1.2.3.3:8080;}
server {    #...省略其余配置    location / {        proxy_pass http://nginxs;    }}

表示当本机接收到请求时，转发到nginxs组内的三台机器。

2. 权重和随机#

# 权重，在server后面添加weight字段，会按比例进行分发。upstream nginxs {    server 1.2.3.1:8080 weight=1;  server 1.2.3.2:8080 weight=2;    server 1.2.3.3:8080 weight=3;}
# 随机，在下方位置处添加random字段。upstream nginxs {    random;    server 1.2.3.1:8080;  server 1.2.3.2:8080;    server 1.2.3.3:8080;}

3. 按时长相应和最少连接数#

fair（公平），按时长相应，需要使用第三方库重新编译才能使用。

# 按时长相应，在下方相应位置处添加fair字段upstream nginxs {    fair;    server 1.2.3.1:8080;  server 1.2.3.2:8080;    server 1.2.3.3:8080;}
# 最少连接数， 在下放位置处添加least_conn字段upstream nginxs {    least_conn;    server 1.2.3.1:8080;  server 1.2.3.2:8080;    server 1.2.3.3:8080;}

4. ip哈希和url哈希#

有些请求是有状态的，后端可能会维持session，上次请求到哪台机器，下一次还得请求到这台机器。

# ip hash，在下方位置处添加ip_hash字段upstream nginxs {  ip_hash;    server 1.2.3.1:8080;  server 1.2.3.2:8080;    server 1.2.3.3:8080;}
# url hash 在下方位置处添加相应字段upstream nginxs{  hash $request_uri consistent;  server 1.2.3.1:8080;  server 1.2.3.2:8080;    server 1.2.3.3:8080;}

故障和备用方案#

有时候服务器可能会请求失败，可以在nginx的配置中进行相关设置，假如所有正在运行的服务器都挂了的话，就开启备用服务器。

代码如下:

upstream nginxs {    server 1.2.3.1:8080;    server 1.2.3.2:8080  max_fails=3 fail_timeout=30s;    server 1.2.3.3:8080  backup;}

max_fails参数代表多少次请求后端失败后悔暂停这个业务点，不再给它发送新的请求。

fail_timeout参数代表当nginx认定这个节点不可用时，暂停多久，默认是10s。

backup参数代表，当所有的节点都挂掉的时候，会把请求打到这个节点上，最后的救命稻草。
